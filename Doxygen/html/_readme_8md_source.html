1 # mon(1)
2 
3  Super-simple monitoring program.
4 
5  `mon` spawned from the needlessly complex
6  frustration that tools like [monit](http://mmonit.com/monit/)
7  provide, with their awkward DSLs and setup. `mon` is written
8  in C, uses less than 400kb of memory, and is incredibly simple
9  to set up.
10 
11 ## Installation
12

14 $ make install
15 ```

20 $ (mkdir /tmp/mon && cd /tmp/mon && curl -L# https://github.com/tj/mon/archive/master.tar.gz | tar zx --strip 1 && make install && rm -rf /tmp/mon)

23 ## Usage
24 
25 ```
26 
27 Usage: mon [options] <command>
28 
29 Options:
30 
31  -V, --version output program version
32  -h, --help output help information
33  -l, --log <path> specify logfile [mon.log]
34  -s, --sleep <sec> sleep seconds before re-executing [1]
35  -S, --status check status of --pidfile
36  -p, --pidfile <path> write pid to <path>
37  -m, --mon-pidfile <path> write mon(1) pid to <path>
38  -P, --prefix <str> add a log prefix
39  -d, --daemonize daemonize the program
40  -a, --attempts <n> retry attempts within 60 seconds [10]
41  -R, --on-restart <cmd> execute <cmd> on restarts
42  -E, --on-error <cmd> execute <cmd> on error
43  -r, --memory write memory usage to logfile
44 
45 ```
46 
47 ## Example
48 
49  The most simple use of `mon(1)` is to simply keep a command running:
50 
51 ```js
52 $ mon ./myprogram
53 mon : pid 50395
54 mon : child 50396
55 mon : sh -c "./example/program.sh"
56 one
57 two
58 three
59 ```
60 
61  You may daemonize mon and disassociate from the term with `-d`:
62 
63 ```js
64 $ mon ./myprogram -d
65 mon : pid 50413
66 ```

68 ## Failure alerts
69 
70  `mon(1)` will continue to attempt restarting your program unless the maximum number
71  of `--attempts` has been exceeded within 60 seconds. Each time a restart is performed
72  the `--on-restart` command is executed, and when `mon(1)` finally bails the `--on-error`
73  command is then executed before mon itself exits and gives up.
74 
75  For example the following will echo "hey" three times before mon realizes that
76  the program is unstable, since it's exiting immediately, thus finally invoking
77  `./email.sh`, or any other script you like.
78 
79 ```js
80 mon "echo hey" --attempts 3 --on-error ./email.sh
81 mon : child 48386
82 mon : sh -c "echo hey"
83 hey
84 mon : last restart less than one second ago
85 mon : 3 attempts remaining
86 mon : child 48387
87 mon : sh -c "echo hey"
88 hey
89 mon : last restart less than one second ago
90 mon : 2 attempts remaining
91 mon : child 48388
92 mon : sh -c "echo hey"
93 hey
94 mon : last restart less than one second ago
95 mon : 1 attempts remaining
96 mon : 3 restarts within less than one second, bailing
97 mon : on error `sh test.sh`
98 emailed failure notice to tobi@ferret-land.com
99 mon : bye :)
100 ```
101 
102  __NOTE__: The process id is passed as an argument to both `--on-error` and `--on-restart` scripts.

104 ## Managing several mon(1) processes
105 
106  `mon(1)` is designed to monitor a single program only, this means a few things,
107  firstly that a single `mon(1)` may crash and it will not influence other programs,
108  secondly that the "configuration" for `mon(1)` is simply a shell script,
109  no need for funky weird inflexible DSLs.
110 
111 ```bash
112 #!/usr/bin/env bash
113 
114 pids="/var/run"
115 app="/www/example.com"
116 
117 mon -d redis-server -p $pids/redis.pid
118 mon -d "node $app/app" -p $pids/app-0.pid
119 mon -d "node $app/jobs" -p $pids/jobs-0.pid
120 mon -d "node $app/jobs" -p $pids/jobs-1.pid
121 mon -d "node $app/jobs" -p $pids/jobs-2.pid
122 mon -d "node $app/image" -p $pids/image-0.pid
123 mon -d "node $app/image" -p $pids/image-1.pid
124 mon -d "node $app/image-broker" -p $pids/image-broker.pid
125 ```
126 
127  I highly recommend checking out jgallen23's [mongroup(1)](https://github.com/jgallen23/mongroup),
128  which provides a great interface for managing any number of `mon(1)` instances.
129 
130 ## Logs
131 
132  By default `mon(1)` logs to stdio, however when daemonized it will default
133  to writing a log file named `./mon.log`. If you have several instances you may
134  wish to `--prefix` the log lines, or specify separate files.
